= RFD 3 s7-in-Elixir vs. a Fine NIF over libriscv, for the BEAM host RFD 2 scopes

xref:../0002/README.adoc[RFD 2] establishes that a BEAM/OTP host -
`weft-warp-burrito`'s GenServer-per-`Machine` + dirty-NIF pattern -
replaces `weft-warp-loop`'s Flow actor as the host for the s7 sandbox
tier, and that true mid-execution *resume* is new work regardless of
what else changes. This RFD answers a question RFD 2 left open by
implication: given that BEAM is now the host, should the *interpreter*
also move onto BEAM natively, or should `libriscv` stay?

Two options:

* *(A) s7-in-Elixir* - reimplement the Scheme semantics `content/*.scm`
  needs as a native Elixir evaluator, running as ordinary BEAM code.
* *(B) Fine NIF + libriscv* - keep s7 compiled to RISC-V and run inside
  `libriscv`, exactly as today, but replace the hand-written
  `erl_nif`-based binding (`c_src/nif/weft_sandbox_nif.cpp`) with one
  generated by `elixir-nx/fine` ("C++ library enabling more ergonomic
  NIFs, tailored to Elixir" - confirmed directly via
  `gh repo view elixir-nx/fine`, not assumed from ADR 0007's mention of
  it).

== Comparison against the four required properties

[cols="1,3,3"]
|===
| Property | (A) s7-in-Elixir | (B) Fine NIF + libriscv

| Sandbox
| Only as strong as "the API never exposes eval" - no isolation
  boundary against bugs *in the interpreter itself*; a buggy Elixir
  evaluator can still misbehave arbitrarily within its own reach.
| Real ISA-level isolation: a bug in s7 or the guest content can only
  corrupt the guest's own emulated address space, never host/BEAM
  memory. Already proven (RFD 1).

| Gas
| Hand-written step-counting in the evaluator loop - fully
  controllable, and arguably *more* meaningful than RISC-V instruction
  count, which is sensitive to how `s7.c` happens to get compiled
  (optimization level, codegen) rather than to script-level work done.
| Already built: `simulate_with<true>(fuel, ...)`, enforced natively by
  `libriscv`.

| Pause
| Free - a GenServer is already idle/inspectable between calls.
| Free - already built (RFD 1/2), same mechanism.

| Resume
| **Easiest path of the two, if built deliberately**: write the
  evaluator so its continuation is an explicit Elixir data structure
  (not the native call stack) - that continuation is then plain data,
  trivially `:erlang.term_to_binary`-serializable, giving checkpoint-
  anywhere resume close to for free.
| **New work, harder**: needs guest-side cooperation (a `(yield ...)`
  primitive) plus `Machine::serialize_to`/`deserialize_from`
  (`c_src/thirdparty/libriscv/lib/libriscv/serialize.cpp`) on the host
  side - buildable (sketched in RFD 2), not done.
|===

== Two considerations that outweigh the table

=== Single source of truth

The guest ELF is Lean4-verified once, in `fanout-core`, against the
three golden vectors RFD 1 documents, and produces byte-identical
output whichever host calls it - Flow/C++ today, BEAM/Elixir per RFD 2
- because both are just issuing a VMCALL into the same
`libriscv::Machine` semantics. Option A creates a second, independent
Scheme evaluator that has to stay in exact semantic agreement with the
C one forever, or the content itself forks between hosts. That is a
standing verification cost, not a one-time porting cost, and it is the
main reason this RFD leans against Option A.

=== Crash isolation is a real gap, orthogonal to this choice

A NIF - Fine-generated or hand-written - runs in-process. A segfault in
the C++/`libriscv` glue takes the whole BEAM node down with it; dirty
schedulers change *scheduling*, not fault isolation. Pure Elixir code
cannot do this. If this risk matters more than the single-source-of-
truth argument above, the actual fix is a third option this RFD names
but does not recommend: run `libriscv`+s7 as a **separate OS process**
over an Erlang Port instead of an in-process NIF - keeps the mature C
interpreter and its ISA-level guest isolation, and gains real OS-level
crash isolation, at the cost of IPC latency per call. This is called
out explicitly so it isn't silently lost: choosing (B) over (A) does
not close this gap, it just declines to fix it here.

== Recommendation

Keep `libriscv` (option B). Migrate `c_src/nif/weft_sandbox_nif.cpp`'s
hand-rolled `erl_nif` marshalling to `elixir-nx/fine` - a binding-layer
cleanup, not an architecture change - and build Resume as RFD 2
sketches (guest-side yield + `Machine` serialize/deserialize). Do not
fork s7's semantics into a second, native-Elixir interpreter: the
maintenance and re-verification cost of keeping two interpreters
byte-identical outweighs Option A's real Resume-ergonomics and crash-
isolation advantages, and the crash-isolation advantage specifically is
available without Option A anyway, via the Port-based alternative
above, if it turns out to be needed.

== Unresolved questions

* Does anything downstream of `weft-warp-burrito`'s guest content
  actually require cross-host (Flow vs. BEAM) byte-identical semantics
  today, or only in `weft-warp-loop`'s Flow-hosted server? If
  `weft-warp-burrito`'s own capabilities (loot/combat/progression) are
  purely local single-player content with no server-side replay
  counterpart, the single-source-of-truth argument is weaker than
  stated above for *this* consumer specifically - though it still holds
  for keeping one guest binary shared across both hosts long-term.
* Is the NIF-vs-Port crash-isolation question worth resolving now, or
  is it acceptable to ship the Fine-based NIF first and revisit Port
  isolation only if a real crash is observed in practice?
* If Resume is built per RFD 2, does its guest-side `(yield ...)`
  primitive change the crash-isolation calculus at all (larger guest
  surface reachable per logical script run), or is it orthogonal?
