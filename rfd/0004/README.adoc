= RFD 4 Migrating taskweft/taskweft and taskweft/nif into weft-warp-burrito

This RFD records a migration actually performed and verified in this
repo, at the user's explicit direction, superseding the narrower
"write a proposal first" path xref:../0003/README.adoc[RFD 3]'s
discussion initially suggested for touching `taskweft`. Unlike RFD 3
(a comparison of two *unbuilt* options), this is the record of a real
move that already happened - closer in kind to
xref:../0001/README.adoc[RFD 1]'s backfill.

== What this is not

This is **not** the s7/libriscv planner-replacement question RFD 3
raised. `taskweft/nif` is migrated here unchanged - still its own
native C++20 HTN planner + HRR implementation, nothing to do with
`s7`/`libriscv`. Whether to ever reimplement taskweft's planner core
on top of the sandbox this repo already has remains exactly as
undecided as RFD 3 left it.

== What moved

Cloned directly from GitHub (`gh` + `git clone`, not the API) and
copied into this repo as two sibling Mix projects:

* `taskweft/` - formerly `github.com/taskweft/taskweft`: the HTN
  planner application itself, including its own bundled MCP server
  (`taskweft mcp [--http [--port N]]`, per its README) - this is what
  makes the migration satisfy "with mcp" without needing the separate
  `taskweft/mcp` or `taskweft/deploy` repos.
* `taskweft_nif/` - formerly `github.com/taskweft/nif`: the native
  C++20 HTN planner + HRR NIF `taskweft/taskweft` depends on.

`taskweft/mix.exs`'s dependency on `taskweft_nif` was changed from a
Hex version constraint to a local path dependency with `override: true`
(required because `taskweft_rebac`, still resolved from Hex, also
declares a `taskweft_nif ~> 0.2.0-dev` constraint that Mix won't
silently let a path dependency satisfy without an explicit override).

=== Excluded from the copy

`thirdparty/gltf/` and `thirdparty/gltf_interactivity/` - vendored
glTF spec example assets, unrelated to the planner/MCP core - were
dropped. Windows' `MAX_PATH` broke the checkout on one deeply nested
example path
(`.../KHR_materials_pbrSpecularGlossiness/examples/convert-between-workflows-bjs/...`)
before this exclusion, the same class of problem RFD 1 already
documents for the RISC-V toolchain's own install path. If anything in
this repo ever needs those spec examples, re-fetch them directly rather
than trying to force the long path through.

=== What did not move

Still external Hex dependencies of `taskweft/`, unchanged:
`taskweft_rebac`, `taskweft_mcp_client`, `oauth_mcp_bridge` (the last
one by taskweft's own original design choice - "deliberately stays its
own repo/Hex package; it's genuinely reusable and not taskweft-
specific," a comment preserved verbatim in the migrated `mix.exs`),
`ex_mcp`, and the rest of the ordinary Hex dependency tree (`rdf`,
`json_ld`, `plug_cowboy`, etc.).

Not migrated at all, and not investigated: `taskweft/deploy`,
`taskweft/mcp`, `taskweft/plans`, `taskweft/rebac`, `taskweft/mcp-
client`, `taskweft/oauth-mcp-bridge`, `taskweft/taskweft-proposals`,
`taskweft/taskweft-mmog-agent`, `taskweft/hex-ops`. Whether any of
these should follow is a separate, later decision.

== Verification performed

* `mix deps.get` resolves cleanly with the path-dependency override.
* `mix compile` builds both the migrated Elixir code and rebuilds
  `taskweft_nif`'s native library from source
  (`priv/libtaskweft_nif.dll`) via its own vendored `Makefile` - not a
  copied prebuilt binary.
* `mix test` - the full migrated suite, 293 tests including
  property-based tests (`propcheck`) and a real MCP client integration
  test against a reference MCP server - passes unchanged in this new
  location.
* A live functional check beyond the test suite:
  `Taskweft.CLI.main(["plan", "priv/plans/domains/blocks_world.jsonld",
  "--problem", "priv/plans/problems/blocks_world_1a.jsonld"])` produces
  a correct real plan
  (`[["a_unstack","a"],["a_putdown","a"],["a_pickup","c"],["a_putdown","c"],["a_pickup","b"],["a_stack","b","a"]]`),
  confirming the relocated native planner actually plans, not just that
  the test suite happens to pass.

== What has not happened

Nothing on GitHub has been touched. `taskweft/taskweft` and
`taskweft/nif` are untouched, unarchived, and still the live
repositories backing the real deployed MCP server (Fly.io) and any
currently-configured MCP client pointing at them. Archiving those
repos is a separate, deliberately deferred step - not part of this
RFD - since it should only happen once whoever depends on the live
`taskweft` MCP endpoint has been repointed at wherever the migrated
copy actually gets served from, which this RFD does not address.

== Unresolved questions

* Where does the migrated `taskweft/` actually get *run* from in
  production - does `weft-warp-burrito` grow its own release/deploy
  step for it (mirroring `taskweft/deploy`'s Fly.io setup, not yet
  migrated), or does something else point at this new location?
* Should the remaining external repos (`rebac`, `mcp-client`, `deploy`,
  `mcp`, `plans`) be migrated too, and if so, in what order - `plans`
  (just JSON-LD content) is the lowest-risk next candidate; `deploy`
  (production hosting/auth) is the highest-risk.
* This repo now contains three independent build systems side by side
  (weft-warp-burrito's own Burrito/NIF/libriscv stack, `taskweft`'s
  Mix/Burrito/NIF stack, `taskweft_nif`'s own Makefile) with no shared
  umbrella project tying them together. Is a plain multi-project
  monorepo the intended end state, or should this become a real Mix
  umbrella project?
